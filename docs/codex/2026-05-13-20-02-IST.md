## Root Cause Found

The local account state for `fenarkhan@gmail.com` was internally inconsistent:

- `auth.users` contained a valid auth account
- `auth.users.raw_user_meta_data` said MFA/passkey was enabled:
  - `hasPasskey: true`
  - `passkeyEnabled: true`
  - `mfaEnabled: true`
- There was no matching row in the app `profiles` table
- There were no matching rows in `passkey_credentials`

Resulting behavior:

- Password login succeeded against Supabase Auth
- Middleware trusted auth metadata and treated MFA as required
- The app attempted to challenge for a passkey
- The passkey challenge could not succeed because the app had no corresponding app-side passkey credential or profile state for that user

This was not a normal browser-only passkey problem. It was a broken local auth/profile synchronization problem.

## Repair Implemented

The user asked to repair the local account while preserving the existing auth user.

Decision taken:

- Preserve the existing Supabase auth user
- Restore app-side consistency
- Leave the account in password-only state after repair
- Clear stale MFA/passkey metadata

Implementation added:

- New script: `scripts/repair-local-account.js`
- New package script: `npm run repair:local-account`

The script performs the following for a target email:

1. Looks up the local `auth.users` record
2. Uses the auth user id as the source of truth
3. Deletes any `passkey_credentials` rows for that user id
4. Creates or updates the matching `profiles` row
5. Resets MFA/passkey fields in `auth.users.raw_user_meta_data`
6. Leaves the existing email/password auth account intact

## Repair Execution Performed
