## Prompt-Writing Rules Used in This Session

These rules should be followed when writing future Kimi K2.6 execution prompts.

### General Kimi prompt standard

Every substantial implementation prompt should instruct Kimi to:

- act like a team of senior software engineers, not a single autocomplete model
- inspect the repo first before coding
- read the relevant PRD(s) first
- understand the exact existing baseline branch and files
- preserve the current architecture and naming patterns
- write production-grade code
- avoid placeholders, fake completeness, and AI slop
- include proper authorization, auditability, tests, and failure handling
- explain files changed, rationale, test coverage, and remaining limitations at the end

### Required structure for good execution prompts

A strong prompt should usually include all of the following:

- repo path
- framework and stack context
- authoritative PRD path(s)
- user-confirmed baseline status
- relevant current branches / stacked branch context
- exact files to inspect first
- what already exists in the codebase
- exact sprint goal and acceptance criteria
- explicit out-of-scope items to prevent phase drift
- engineering requirements:
  - no AI slop
  - no shallow patches
  - preserve security
  - respect current architecture
- recommended implementation plan
- testing requirements
- deliverables
- required output format for Kimi’s completion report

### Rules for avoiding bad prompts

Do not write vague prompts like:

- “implement this sprint fully”
- “make it production ready” without specifics
- “add the feature” without branch, file, PRD, and baseline context

Do not allow prompt drift into later phases. Explicitly ban unrelated work.

Do not assume merged state unless the user confirmed approval/merge or repo inspection proves it.

### Mailbox Sprint 4.3 review outcome

The PR is close, but not yet ready for approval. Three meaningful issues were found.

#### 1. Missing Prisma migration for new mailbox audit enum values

Problem:

- `prisma/schema.prisma` adds:
  - `THREAD_READ`
  - `THREAD_UNREAD`
  - `THREAD_FLAGGED`
  - `THREAD_UNFLAGGED`
- but no new migration directory was added under `prisma/migrations`

Risk:

- runtime audit writes can fail on an existing DB because the DB enum may not match the schema/client

#### 2. `unarchive` currently destroys prior workflow status

Problem:

- `unarchive` in `src/lib/mailbox/thread-action-service.ts` always restores status to `OPEN`

Why that is wrong:

- if a thread was previously `PENDING` or `CLOSED`, archive/unarchive changes its workflow state unexpectedly
- this violates the Sprint 4.3 PRD expectation that mailbox state remains consistent after actions

#### 3. Thread-list quick action does not support unarchive

Problem:

- thread-list quick action in `src/app/app/mailbox/mailbox-thread-list.tsx` always invokes `archive`
- it does not toggle to `unarchive` for already archived rows
- the reading pane header does support archive/unarchive correctly, so the UI is inconsistent
