## Changes Made This Session (Jun 18, 2026)

### A. Command Bar "+ New" Dropdown Modals Fix

**Problem:** Clicking "+ New" → "New message" / "New channel" / "New group" in the command bar did nothing. The `onClick` handlers only closed the dropdown but never opened creation modals.

**Files changed:**

1. **`src/app/app/messaging/messaging-command-bar.tsx`**
   - Added callback props: `onCreateMessageClick`, `onCreateChannelClick`, `onCreateGroupClick`
   - Each dropdown item now calls its respective callback + closes the menu

2. **`src/app/app/messaging/messaging-workspace.tsx`**
   - Added hoisted state: `showCreateChannel`, `showCreateGroup`, `showCreateDM`
   - Wired `MessagingCommandBar` callbacks to open the appropriate modal
   - Renders `MessagingChannelCreate`, `MessagingGroupCreate`, `MessagingDMCreate` as overlays inside the workspace shell, connected to `createConversation`
   - List components receive `onRequestCreate` prop for their header "+" buttons

3. **`src/app/app/messaging/messaging-conversation-list.tsx`**
   - Removed local `showCreate` state from `ChannelConversationList`, `DMConversationList`, `GroupConversationList`
   - Removed local conditional rendering of create modals (moved to workspace)
   - Removed imports of create modal components
   - Added `onRequestCreate` prop; header "+" buttons and "Start a new message…" button call this instead of local state
   - Removed unused `onCreate*`/`creating*` props (interface + destructuring)

### B. Left Panel "Loading conversations…" Flash Fix

**Problem:** Sending a message triggered `refreshList()` which called `setLoading(true)`, causing the conversation list to unmount and show "Loading conversations…" briefly.

**Files changed:**

1. **`src/app/app/messaging/lib/use-conversation-list.ts`**
   - `load` now accepts `options?: { silent?: boolean }`
   - `setLoading(true)` only fires when `!options?.silent`

2. **`src/app/app/messaging/messaging-workspace.tsx`**
   - All 8 user-action-triggered `refreshList()` calls changed to `refreshList({ silent: true })`:
     - After sending a message
     - After replying in a thread
     - On manual refresh detail
     - Task create onSuccess
     - Channel/Group/DM creation onSuccess
     - Portal create from prompt onSuccess
     - Mark-read effect side effect

### C. User Display Name Truncated UUID Fix

**Problem:** Message author names displayed as a truncated UUID (e.g., `87224d19`) because the backend's `ConversationDetail` never included `participantProfiles`, so the frontend fell back to `msg.authorId.slice(0, 8)`.

**Files changed:**

1. **`src/lib/messaging/read-shapes.ts`**
   - Added `ParticipantProfile` interface (`userId`, `name`, `avatarInitials`)
   - Added `participantProfiles?: ParticipantProfile[]` to `ConversationDetail`
   - Added `profiles` to `ConversationDetailInput`
   - `toConversationDetail` maps profiles with derived initials from name

2. **`src/lib/messaging/read-models.ts`**
   - `getConversationDetail` extracts `userIds` from participants, fetches profiles from `db.profile.findMany` in parallel with reactions/attachments/mentions, and passes to `toConversationDetail`

3. **Test files** (`messaging-sprint-2-3.test.ts`, `messaging-sprint-3-1.test.ts`, `messaging-sprint-3-3.test.ts`)
   - Added `profile: { findMany: makeFn() }` to mock DB setup
